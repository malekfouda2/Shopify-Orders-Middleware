ORDER LIFECYCLE BEHAVIOR
==================================================

You must support the full order lifecycle in v1.

A. ORDER CREATION
When a Shopify order is created:
- create a corresponding Tabliya order using POST /api/orders

B. ORDER UPDATE
When a Shopify order is edited or updated:
- find the corresponding Tabliya order
- update it using PUT /api/orders/<id>
- sync changed buyer info, address, delivery date, payment status, payment method, notes, and any fields supported by Tabliya
- if Tabliya order item updates are not directly supported in a single update endpoint, document the limitation clearly and implement the best safe behavior possible
- preserve traceability between Shopify order and Tabliya order

C. ORDER CANCELLATION
This is important and mandatory.

When a Shopify order is cancelled:
- reflect that in Tabliya
- use the most correct behavior supported by the attached API documentation

Implement cancellation logic safely:
1. First preference:
   - if Tabliya supports updating order status to a cancellation-like state via PUT /api/orders/<id>, use that
2. If cancellation status is not supported but delete is the intended pattern:
   - use DELETE /api/orders/<id>
3. If neither is safe enough from the docs alone:
   - mark for manual review and log prominently

Also:
- preserve cancellation reason, cancelled timestamp, and Shopify references in notes/logs when possible
- prevent repeated cancellation actions from causing inconsistent state

D. OPTIONAL INVOICE FOLLOW-UP
If the existing Tabliya flow expects invoices to mirror order payment/address changes, add optional support for:
- POST /api/invoices
- PUT /api/invoices/<id>

But keep invoice sync behind a config flag if needed.
Order sync is the main priority.

==================================================
FIELD MAPPING RULES
==================================================

Transform Shopify orders into Tabliya order payloads.

Use these mapping rules:

Tabliya order_number:
- use a unique integration-safe value like:
  SHOPIFY-<shopify_order_id>

Also include the original Shopify order name/number in notes.

Tabliya order_date:
- use Shopify created date converted to YYYY-MM-DD

Tabliya delivery_date:
- first try to get it from Shopify note attributes / cart attributes / metafields / tags if available
- otherwise use a configurable fallback:
  created_at + DEFAULT_DELIVERY_OFFSET_DAYS

Tabliya buyer_name:
- use shipping full name first
- otherwise billing full name
- otherwise customer name
- otherwise fallback to email

Tabliya payment_status:
Map Shopify status carefully:
- paid -> paid
- partially paid -> partial
- pending/authorized/unpaid-like -> unpaid

Tabliya amount_paid_partial:
- set when payment_status = partial
- compute from Shopify amounts if possible

Tabliya street_name and house_number:
- parse from Shopify shipping address line 1
- best-effort parser for common EU/Dutch address styles
- if parsing fails, store full line in street_name and use configurable fallback for house_number
- log warning

Tabliya zip_code:
- use shipping postal code

Tabliya payment_method:
- resolve using PaymentMethodMapping

Tabliya notes:
Include:
- Shopify order name
- Shopify order ID
- customer note
- cancellation note if any
- refund/cancellation context if relevant
- original payment gateway label
- discount info if not cleanly representable
- delivery notes
- anything important for traceability

Line item mapping:
For each Shopify line item:
- resolve product mapping by Shopify variant ID
- map quantity directly
- calculate discount_percent if possible
- if not possible, set discount_percent = 0 and add discount details in notes
- include SKU/variant traceability in item notes

If any item is unmapped:
- do not sync the order
- log exact unmapped items
- mark as failed/manual review
- allow replay later
